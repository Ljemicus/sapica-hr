## Legacy resolution notes

- `provider_applications` does **not** exist live, so every remaining codepath touching it is ghost/legacy and must be removed or rewritten in Cycle 9.
- `profile_roles` is the only live role source; any read from `profiles.role` or `user_metadata.role` is non-canonical and belongs to Cycle 7.
- `bookings` is retained, but its **column model in code is stale**; Cycle 8 is the required rewrite before booking/payment trust can be considered canonical.
- `payments` owns Stripe session identifiers live (`stripe_checkout_session_id`), not `bookings`.
- `messages` exists live only in the conversation-scoped schema (`conversation_id`, `sender_profile_id`); current peer-to-peer message code is a legacy abstraction that must be rewritten before it is trustworthy.
- Trainer tables are retained because live content exists, but they are not the canonical provider capability model; canonical provider reads/writes should flow through `providers` + `provider_trainer_settings`.
